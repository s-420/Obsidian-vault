## 一、今日学习内容

### 1. Redis Stream 异步任务处理

**核心概念**：
- Redis Stream 是 Redis 5.0 引入的持久化消息队列，优于 Redis Pub/Sub 和 BLPOP
- 消费者组机制支持多消费者负载均衡
- ACK 确认机制确保消息不丢失
- 支持阻塞读取，减少 CPU 轮询消耗

**异步处理流程**：
```
上传简历（同步）
    ↓
发送任务到 Stream（同步）
    ↓
立即返回 PENDING 状态
    ↓
后台消费者处理（异步）
    - 标记 PROCESSING
    - AI 分析
    - 标记 COMPLETED/FAILED
```

**关键组件**：
- `AnalyzeStreamProducer` - 生产者，发送任务
- `AbstractStreamProducer` - 基类，统一发送逻辑
- `AnalyzeStreamConsumer` - 消费者，处理任务
- `AbstractStreamConsumer` - 基类，统一消费逻辑

**重试机制**：
- 最大重试次数：`MAX_RETRY_COUNT`
- 失败后重新入队，超过次数标记为 FAILED

### 2. 文件去重机制

**实现原理**：
```
文件去重 = SHA-256 哈希 + 数据库查询
```

**流程**：
1. 用户上传文件
2. 计算文件 SHA-256 哈希值
3. 查询数据库是否存在相同哈希
4. 存在 → 返回历史分析结果，更新 access_count
5. 不存在 → 继续正常上传流程

**优势**：
- 节省存储空间
- 节省 AI 分析成本
- 提升响应速度
- 数据一致性

### 3. JPA vs MyBatis/MyBatis-Plus

| 维度 | Spring Data JPA | MyBatis/MyBatis-Plus |
|------|-----------------|---------------------|
| 设计哲学 | 面向对象（DDD） | 面向 SQL |
| 单表效率 | 高 | MP 同样高 |
| 复杂查询 | 弱/繁琐 | 强 |
| 性能优化 | 容易踩坑 | 完全可控 |
| 学习曲线 | 陡峭 | 平缓 |
| 数据库移植性 | 强 | 弱 |

**选型建议**：
- 欧美外企/传统企业 → JPA/Hibernate
- 国内互联网/ToC项目 → MyBatis-Plus
